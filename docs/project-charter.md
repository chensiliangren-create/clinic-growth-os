# Clinic Growth OS プロジェクト憲章

## ミッション（我々の存在意義）

整骨院・治療院を経営する個人事業主に対して、
Webサービスと実行支援を通じて、
集客・予約・リピート・経営改善の課題を解決する。

院長が**施術と患者対応に集中できる状態**を作ることが、我々の使命である。

---

## ターゲット（誰を助けるのか）

### 主な対象
- 整骨院・接骨院・整体院・治療院などの個人事業主
- 鍼灸院、マッサージ店など、施術系ビジネス全般

### 初期ターゲット層
- **年齢**: 30～50代
- **規模**: 1店舗経営
- **スタッフ**: 1～5名（多くは施術者＋受付兼任）
- **抱える課題**: Google口コミ・LINE・SNS・予約管理・リピート率に課題を感じている層

### 除外ターゲット
- 大規模展開済みのチェーン店
- 既に外部マーケティング支援を受けている層
- 抽象的なノウハウ販売に関心のみある層

---

## 提供内容（何を売るのか）

### 売る価値
**ノウハウではなく、具体的なサービスである。**

「方法を教える」のではなく、「こちらが一緒に実行し、成果が出る状態まで持っていく」ことを価値にする。

### 初期提供サービス

1. **Google口コミ管理と改善**
   - Googleビジネスプロフィール最適化
   - 口コミ獲得の導線設計
   - 低評価への対応サポート

2. **予約・リピート導線の設計**
   - LINE公式アカウント運用設計
   - リマインド自動化テンプレ
   - 再予約・定期来院の仕組み

3. **簡易KPI管理**
   - 月間新規数・リピート率の可視化
   - ROI測定（集客費用対効果）
   - 院長向けダッシュボード

4. **Webサイト・LP改善**
   - 初期Webサイト制作またはリニューアル
   - 新規患者向けLPの最適化
   - モバイル対応

5. **ヒアリングに基づく改善提案**
   - 定期的な経営診断
   - 数値分析による改善案
   - 実装支援とフォロー

## 提供しない内容（何を売らないのか）

### 提供しないもの
1. **情報商材・講座だけの商品**
   - 「自分で頑張ってください」は提供しない
   - ノウハウ本や講義だけの販売はしない

2. **抽象的なコンサルティング**
   - 「考える力を磨こう」など精神論的なアドバイス
   - 実装が伴わないコンセプト提案

3. **完全自動化・AIに頼る見せかけのサービス**
   - 最初から全自動AIを約束しない
   - 人力の伴走が不可欠な段階を隠さない

4. **短期バイラル・即座に売上3倍などの虚偽**
   - 現実的でない成果約束
   - 院長のビジネス規模を無視した施策提案

## 成長戦略（どうやって大きくなるのか）

### フェーズ1: 事例化（今、市場調査～3ヶ月）
1. 30人インタビューで共通課題を発見
2. ペルソナと優先課題を特定
3. 1院への深い伴走支援
4. 実績データの収集と事例化

### フェーズ2: 横展開（3～6ヶ月）
1. 事例をベースに営業テスト
2. 3院、5院、10院へ支援対象を増やす
3. 共通テンプレの抽出と改善
4. 実運用で得られたノウハウの言語化

### フェーズ3: テンプレート化（6～12ヶ月）
1. 共通業務をテンプレート化
2. 院長セルフサービス向けのドキュメント作成
3. 一部業務をスタッフやAIに移譲
4. 提供単価と効率のバランス調整

### フェーズ4: Webサービス化（12ヶ月～）
1. 簡易管理ツール（Google Sheets連携など）のWebサービス化
2. 口コミ自動返信、予約リマインドなどの自動化
3. KPI可視化ダッシュボード
4. 複数院での導入を想定した拡張性

### フェーズ5: OS化（最終ビジョン）
1. **Clinic Growth OS**：治療院向けの一元管理プラットフォーム
   - 集客管理
   - 予約・スケジュール管理
   - 口コミ・評判管理
   - LINE・メール自動化
   - リピート率管理
   - 売上・ROI分析
2. 複数ビジネス（SaaS + コンサル + 運用代行）の組み合わせ

### 原則：最初から大きなSaaSを作らない
- 人力 → AI・テンプレ → 半自動化 → フルサービス の順で進める
- 各段階で院長からのフィードバックを反映
- スケーラビリティより、まずは実効性を優先

## ビジョン（長期目標）

### 3年後の理想像
整骨院・治療院の院長が、
集客や経営管理に追われず、
**患者と施術に集中できる世界**を作る。

### 10年後の目指す状態
Clinic Growth OSが、整骨院・治療院業界の標準インフラになっている。

院長が、
集客から経営分析までを、
一つのシステムで管理でき、
スタッフとの情報共有も円滑になっている世界。

## 判断基準（意思決定の原則）

### 開発・営業・施策判断の軸

このプロジェクト憲章が、すべての判断の基準になります。

#### 意思決定の質問リスト

1. **これは院長の「実務課題」を解決するか？**
   - YES → 検討対象
   - NO → 提供しない（ノウハウ販売・講座に陥らない）

2. **院長が「すぐに実行できる」形で提供できるか？**
   - YES → 検討対象
   - NO → テンプレ化・自動化できるまで待つ

3. **我々が一緒に実行支援できるか？**
   - YES → 検討対象
   - NO → 院長セルフサービス化まで待つ

4. **小規模院（スタッフ1～5名）に適用できるか？**
   - YES → 検討対象
   - NO → 後発フェーズへ

5. **成果が測定可能か（新規数、リピート率、売上など）？**
   - YES → 検討対象
   - NO → 曖昧な見せかけの可能性あり、避ける

6. **短期（3ヶ月以内）に成果を出せるか？**
   - YES → 優先度高
   - NO → 中期計画へ

#### 営業・LP文言の確認リスト

- [ ] 虚偽や誇張はないか？ → 現実的な数字のみを使用
- [ ] 「やり方だけ教える」になってないか？ → 実行支援を明示
- [ ] AI・自動化を誇大表現していないか？ → 人力の段階を明記
- [ ] 「月に100万円の売上UP」など現実的でない約束をしていないか？
- [ ] 院長のストレスを減らす提案になっているか？

#### 市場調査・インタビューの確認リスト

- [ ] 院長の言葉で課題を記録しているか？（こちらの想像で解釈していないか）
- [ ] 事例ベース（実際に困っている）を集めているか？
- [ ] 個別差異と共通パターンを分けているか？
- [ ] 調査結果が、次のサービス設計に反映されるか？

#### MVP・サービス設計の確認リスト

- [ ] 1院への伴走が前提になっているか？
- [ ] テンプレート化・自動化は後付けしない前提か？
- [ ] 測定可能な成果目標が明確か？
- [ ] 最初から「完全SaaS」を目指していないか？

## このプロジェクト憲章の使い方

### ドキュメント間の関係性

```
project-charter.md（大方針・判断基準）
    ↓
market-research-approach.md（調査方法・質問）
problem-analysis.md（仮説集・共通課題）
service-concept.md（サービス内容・提供方法）
mvp-plan.md（初期実装・1院支援）
    ↓
sales-script.md（実営業・見込み客接触）
site/index.html（LP・協力院募集）
```

### 毎月の確認リスト

1. **憲章ブレがないか確認する**
   - 営業トークが「ノウハウ販売」に傾いていないか
   - 提供内容が「実行支援」ベースか

2. **新しい施策は憲章の判断基準を通したか**
   - 院長の実務課題を解決するか
   - すぐに実行できる形か
   - 成果が測定可能か

3. **インタビュー結果を反映したか**
   - 市場調査のフィードバックが施策に活かされているか
   - 仮説が検証されたか

4. **スコープクリープ（範囲の膨張）を防ぐ**
   - 院長側の「要望」がすべてではない
   - 我々の理想と院長の課題が一致しているか
